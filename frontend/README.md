# JM Test Center — Admin Frontend

基于 **Vue 3**、**Vite 5**、**TypeScript** 的管理端；**Element Plus**（中文语言包）、**Pinia**、**Vue Router**；接口通过 **axios** 封装在 `src/api/`，请求路径以 **`/api`** 开头，由开发服务器或网关转发到后端。

---

## 前端服务在整体中的角色

| 内容 | 说明 |
|------|------|
| **运行形态** | 浏览器访问本前端页面；业务数据一律走后端 REST API（`/api/v1/...`），本仓库不内置业务服务端。 |
| **鉴权** | `src/api/http.ts` 中 axios 拦截器为请求附加 `Authorization: Bearer <token>`；响应 **401** 时清登录态并跳转登录页。 |
| **开发时联调** | 见下文 **`npm run dev`**：`vite.config.ts` 将 **`/api`** 代理到本机后端，避免浏览器跨域。 |
| **环境变量** | 根目录 **`.env.development`** 在 **`npm run dev`** 时注入（如 `VITE_APP_NAME`）；仅 `VITE_` 前缀变量会暴露给前端代码。 |

---

## npm 脚本说明（按功能）

在 **`frontend`** 目录执行，脚本定义见 **`package.json`**。

### `npm run dev` → `vite`

| 作用 | 说明 |
|------|------|
| **启动开发服务器** | 使用 Vite 启动本地开发环境，默认 **`http://127.0.0.1:5173`**（以终端输出为准）。 |
| **热更新（HMR）** | 修改 `.vue` / `.ts` 等源码后，页面模块热替换，无需整页刷新即可看到效果。 |
| **加载开发环境变量** | 读取 **`.env.development`**（若存在），例如 `VITE_APP_NAME`。 |
| **API 反向代理** | 按 **`vite.config.ts`** 中 `server.proxy`：将以 **`/api`** 开头的请求转发到 **`http://127.0.0.1:8080`**。前端代码里 axios 使用相对路径（如 `/api/v1/auth/login`），浏览器发往当前页面 origin，由 Vite 代发到后端。 |
| **适用场景** | 日常开发、与后端联调、调试路由与 UI。 |

**注意**：后端需已启动，且监听端口与 `proxy["/api"].target` 一致；若后端端口不是 **8080**，请修改 **`vite.config.ts`** 中的 `target`。

---

### `npm run build` → `vue-tsc -b && vite build`

分两步顺序执行，任一步失败则整个命令失败。

| 步骤 | 命令 | 作用 |
|------|------|------|
| 1 | **`vue-tsc -b`** | 使用 **vue-tsc** 对 Vue SFC + TypeScript 做**类型检查**（基于 `tsconfig.json` 等）。有类型错误时**不会**进入下一步，避免带病打包。 |
| 2 | **`vite build`** | 类型检查通过后，执行 **生产构建**：打包、压缩资源，输出到项目根下 **`dist/`** 目录（静态 HTML/JS/CSS，可直接由 Nginx 等托管）。 |

| 适用场景 | 说明 |
|----------|------|
| **发版 / 部署前** | 生成可部署的静态产物。 |
| **CI** | 在流水线中执行 `npm ci && npm run build` 做构建校验。 |

**生产环境请求后端**：构建产物中的接口仍为相对路径 `/api/...`，部署时需由 **同源网关** 或 **反向代理** 将 `/api` 转到真实后端；开发用的 Vite `proxy` **不会**带到 `dist` 里。

---

### `npm run preview` → `vite preview`

| 作用 | 说明 |
|------|------|
| **本地预览生产包** | 在本地起一个静态服务，**默认**预览 **`dist/`** 的内容（端口以终端为准，常见为 **4173**）。用于确认 `build` 结果、资源路径、首屏是否正常。 |
| **与 `dev` 的差异** | 预览的是**已构建**的静态文件，无源码 HMR；当前 **`vite.config.ts` 未为 `preview` 单独配置 `proxy`**，若在此模式下页面需要调用 `/api`，需自行保证浏览器能访问到后端（例如同域部署、或为 `preview` 增加与 `server` 相同的代理）。日常接口联调更推荐 **`npm run dev`**。 |

| 适用场景 | 说明 |
|----------|------|
| **构建验收** | `build` 之后本地打开产物看一眼。 |
| **排查打包问题** | 对比 dev 与生产包行为差异时有用。 |

---

## 脚本一览表

| 脚本 | 底层命令 | 一句话 |
|------|----------|--------|
| **`dev`** | `vite` | 开发服务器 + HMR + 读 `.env.development` + **`/api` 代理到后端**。 |
| **`build`** | `vue-tsc -b && vite build` | 先 **TS/Vue 类型检查**，再 **打包到 `dist/`**。 |
| **`preview`** | `vite preview` | **本地预览 `dist/`**，不等同于带代理的完整联调环境。 |

## 环境要求与安装

- **Node.js** 建议 18+（与 Vite 5 一致）。
- 首次克隆后在 **`frontend`** 目录执行：

```bash
npm install
npm run dev
```

浏览器打开终端提示的地址；登录等请求经 **`/api`** 代理到后端（请先启动后端，见仓库 **`backend/README.md`**）。
